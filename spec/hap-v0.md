# HAP · Hiring Agent Protocol v0.1 (draft)

> **Status**: Draft. Comments → [GitHub issues].
> **Versioning**: v0.x = breaking changes allowed. v1.0 = stable.
> **Base layer**: Google's [A2A (Agent2Agent) Protocol](https://a2a-protocol.org/). HAP is an **A2A profile** for the hiring use case — not a competing protocol.
> **License**: Apache 2.0 (spec) + MIT (reference impls).

## Why

Recruitment today flows through resume PDFs and a handful of ATS giants. Both make AI-mediated hiring brittle:

- **Resumes are static.** Candidates over- or under-claim; AI generators fake them; HR can't ask follow-ups.
- **ATS are walled.** Greenhouse / Lever / Workday own the data. Independent tooling can't compose.

When both sides of the hiring conversation are AI agents — and they will be — they need a **standard interview protocol** to talk to each other. HAP is that protocol.

**One sentence**: `HAP` lets a candidate-agent and an HR-agent run a structured interview without either side handing over a full resume or full job description upfront.

## Architecture

```
           ┌──────────────────┐
           │   HR-agent       │   ◀── holds the JD, asks questions
           │ (any vendor)     │
           └────────┬─────────┘
                    │ A2A
                    ▼
           ┌──────────────────┐
           │  HAP Skill     │   ◀── this spec (A2A Skill / AgentCard)
           │  message types   │
           └────────┬─────────┘
                    │ A2A
                    ▼
           ┌──────────────────┐
           │ Candidate-agent  │   ◀── holds candidate evidence,
           │ (any vendor)     │       answers from public traces +
           └──────────────────┘       declared self-claims
```

No central server. No database. No vendor lock. Each side runs its own agent on its own infrastructure. Either side can use any LLM (or no LLM). The wire format is A2A.

## Design principles

1. **Federated, not blockchain.** Like email (SMTP) and Mastodon (ActivityPub) — open protocol, peer-to-peer message routing, no on-chain claims. We considered W3C Verifiable Credentials and rejected them as premature.
2. **Evidence over claim.** Candidate-agents respond with URLs to public traces (GitHub commit SHA, talk recording URL, paper DOI), not unverifiable assertions.
3. **Progressive disclosure.** Candidate doesn't dump a full profile. HR-agent asks; candidate-agent answers with relevant evidence only. BitTorrent-style piece-wise download.
4. **A2A profile, not a new protocol.** HAP messages MUST be valid A2A messages. We just standardize the *content* schemas inside A2A's envelope.
5. **BYO LLM optional.** Reference runtimes work with any LLM (OpenAI, Anthropic, local). For users without a key, hap.dev offers a hosted free-tier proxy with daily quota.

## Roles

| Role | What it holds | What it does |
|---|---|---|
| **HR-agent** | One or more JDs | Drives the interview: opens a session, asks structured questions, may request live elaboration |
| **Candidate-agent** | A candidate's identity + pointers to public artifacts (GitHub user, blog URL, talks, etc.) + self-declared preferences | Responds to HR-agent queries with evidence references and short explanations. Decides what to disclose. |
| **Tracker** (optional) | A directory of agents available to receive HR-agent contact (similar to BitTorrent tracker) | Lightweight discovery layer. Not required for direct-known-URL contact. |

## Message types (v0.1)

All messages are A2A `Message` objects with a HAP-specific `kind` field in the metadata. The body schemas below describe the `parts` content.

### `hap.session.open`

HR-agent → candidate-agent. Opens a HAP session for a specific JD.

```json
{
  "kind": "hap.session.open",
  "session_id": "h_01HXY...",          // ULID generated by HR-agent
  "jd": {
    "title": "Senior Backend Engineer · Payments",
    "summary": "≤500 chars",
    "must_have": ["5+ yrs backend", "payments domain", "Go or Rust"],
    "nice_to_have": ["OSS contributions", "public talks"],
    "domain": "payments",              // optional, used by routing
    "external_url": "https://..."      // optional, full JD page
  },
  "from": {
    "agent_url": "https://hr.company.com/.well-known/agent.json",
    "company": "Acme Corp",            // optional
    "human_contact": "hiring@..."      // optional, for follow-up
  }
}
```

Candidate-agent decides to accept (responds with `hap.session.accept`) or decline (`hap.session.decline` with optional reason).

### `hap.ask`

HR-agent → candidate-agent. Asks a specific question.

```json
{
  "kind": "hap.ask",
  "session_id": "h_01HXY...",
  "question_id": "q1",
  "ask": {
    "type": "open" | "scored" | "yes_no" | "evidence_request",
    "prompt": "What's your experience with idempotency systems at high RPS?",
    "evidence_preference": ["github", "talk", "blog"]   // optional
  }
}
```

### `hap.answer`

Candidate-agent → HR-agent. Answers a question.

```json
{
  "kind": "hap.answer",
  "session_id": "h_01HXY...",
  "question_id": "q1",
  "answer": {
    "text": "I designed Stripe's idempotency key system for the payment routing service ...",
    "evidence": [
      {
        "type": "github_commit",
        "url": "https://github.com/.../commit/abc123",
        "note": "PR introducing the key normalization"
      },
      {
        "type": "talk",
        "url": "https://youtu.be/...",
        "venue": "GopherCon 2024",
        "title": "Designing Idempotent Payment APIs"
      }
    ],
    "confidence": "high" | "medium" | "low",
    "decline_reason": null   // set if candidate declines to answer
  }
}
```

### `hap.session.close`

Either side. Closes the session.

```json
{
  "kind": "hap.session.close",
  "session_id": "h_01HXY...",
  "outcome": "fit" | "no_fit" | "unclear" | "candidate_declined",
  "summary": "≤500 chars (rationale, no PII)",
  "next_step": "human_follow_up" | "schedule_human_interview" | "archive"
}
```

## Evidence types (registry)

`evidence.type` is an open registry. v0.1 ships with:

| Type | URL must point to | Verification hint |
|---|---|---|
| `github_user` | `github.com/<user>` | account age, public repos, commit graph |
| `github_repo` | `github.com/<user>/<repo>` | stars, language, last commit |
| `github_commit` | `github.com/.../commit/<sha>` | commit author email matches candidate |
| `github_pr` | `github.com/.../pull/<n>` | merged status |
| `talk` | conference page or video URL | conference name + speaker name appear |
| `blog_post` | personal site or platform URL | author byline matches candidate |
| `paper` | arxiv.org / DOI URL | author list contains candidate |
| `package` | npm / pypi / crates.io URL | maintainer = candidate |
| `linkedin` | linkedin.com/in/... | (low-trust: auth-walled, hard to verify) |
| `personal_site` | candidate-owned domain | DNS + content matches name |
| `oss_maintainer` | reachable repo | maintainers list contains candidate |
| `email_domain` | mailto: with company domain | optional MX check |

Verifier impls live in `packages/verifiers/`. Anyone can add a new type; the registry is community-extended.

## Disclosure model

A candidate-agent decides which evidence to disclose based on the JD context. Rules of thumb (informative, not normative):

- Default to disclosing **public-by-default** evidence (GitHub user, OSS repos, public talks).
- Withhold **private/PII** evidence (linkedin, email_domain, personal_site that contains contact) unless HR-agent asks for it AND session has been accepted.
- Always include `confidence` so HR can weight low-confidence claims appropriately.
- Refusing to answer is fine: `decline_reason` is informative, not pejorative.

This is BitTorrent-style **piece-wise disclosure**: the HR-agent only sees the parts it queries about, not the full resume.

## Agent cards

HAP agents publish an A2A `AgentCard` at `.well-known/agent.json` per A2A standard. Additional HAP fields:

```json
{
  "name": "...",
  "skills": ["hap.v0"],            // declares HAP support
  "HAP": {
    "role": "candidate" | "hr",
    "supported_versions": ["v0.1"],
    "supported_evidence_types": ["github_user", "github_repo", "talk", ...],
    "rate_limit": { "per_day": 100, "per_hour": 20 },
    "human_contact": "optional@..."  // for high-stakes follow-up
  }
}
```

## Privacy model

- **No central storage.** Each agent keeps its own data. The protocol relays messages but doesn't persist them.
- **No identity broker.** Agents authenticate each other via A2A's standard (HTTPS + optional bearer auth). Candidate identity is asserted by the agent runner, not enforced cryptographically.
- **GDPR compliance** is the responsibility of each agent operator. Reference impls include audit-log toggles.

## Versioning

- v0.x: protocol is unstable, breaking changes possible.
- v1.0: schemas stabilize. Backward-compatible additions only.
- All messages MUST include `HAP_version` in the A2A `metadata.hap.version` field. Clients SHOULD reject mismatched majors.

## Non-goals

- Replacing ATS — HAP is the *interview channel*, not the application tracking system.
- Identity verification — that's KYC providers' job; HAP doesn't validate that an agent's claimed candidate is real.
- Job posting marketplace — HAP doesn't list JDs; HR-agents do.
- Scoring or ranking — that's the consuming app's job. HAP only standardizes the message format.

## FAQ

**Q: Why not just use MCP?**
MCP is agent ↔ tool. HAP is agent ↔ agent. A2A is the right layer.

**Q: Why not blockchain credentials?**
Adoption is 3–5 years out. We don't want to be a dead-on-arrival spec waiting for an ecosystem.

**Q: Will candidates trust their agent with their real data?**
That's the candidate's choice. Reference impls are self-host by default. Hosted convenience tier is optional and uses a `do not log` policy by default.

**Q: How do I prevent spam?**
The HR-agent's `from.agent_url` must serve a valid AgentCard, signed by HTTPS. Spam is a federation-policy problem (like email): blocklists, rate limits, reputation services on top of the protocol.

**Q: Where can I see this running?**
Reference demo at [hap.dev/demo] — watch an HR-agent interview a candidate-agent in 60 seconds. Both runtimes are open source.
