JD-Match 退化原因（三个层次）：
  1. 合成 → GitHub README：文档格式从「简历」变成「bio」，缺少职位/公司/年份结构
  2. GitHub README → 真 PDF：实际生产用户的简历**比合成的多样得多**——使用模板、OCR 噪声、混合格式、跨技能栈

3 · 关键 takeaway · 给投资人 / 客户的「真话」

❌ 不能说：
  • "jd-match LLM 比 bag-of-words 提升 17%" — 真实数据上是 0%
  • "零误报" — 真实数据上是 5% 误报率
  • "F1 0.93+" — 真实数据上 0.86
✅ 可以说：
  • "AI 简历检测 F1 0.86 在真实 livecareer 简历数据上" — 这个数字有公开数据支撑
  • "adversarial AI 召回率 50% — 50% 总比规则版 0% 强"
  • "AI 检测能力跨多个数据集稳定（0.93 / 0.95 / 0.86）"
  • "jd-match LLM 在格式良好的简历上比 bag-of-words 更精细（synthetic 数据 +19%），但实际生产价值还需要在真实候选人简历上验证"

4 · 这次 eval 暴露的真问题

  1. JD-match LLM prompt 需要重做：它对"完美简历格式"的依赖太强。需要训练 prompt 处理 OCR 噪声、多语言、混合格式的真实简历。
  2. 真人误报问题：3 个真人简历被判 AI。需要看具体内容，可能是 livecareer 模板简历（人写但用模板）触发了 AI 检测。需要 prompt 加 "templated by human ≠ AI" 规则。
  3. Ground truth 自动化标注不可靠：用 NER skills 自动分桶可能跟"应该被推荐到 X JD"不完全一致。需要人工 review 几十个样本来 calibrate 标签。
  4. Adversarial AI 在真实分布下更难抓：召回率 50% 是非常诚实的数字。需要继续投入 prompt + 多模型 ensemble。

5 · Methodology & Caveats

  • 真 PDF 数据来源：HuggingFace Mehyaar/Annotated_NER_PDF_Resumes（MIT licensed），5,029 份 CV，原始 PDF 经 OCR 后含 NER 标注。从中采样 60 份（20 backend / 20 frontend / 20 data-ml），按 NER skills clarity 选 top-margin。
  • 同 AI 集：3 次 eval 用同一批 30 份 GPT-5.5 生成 AI 简历，保证可比性。
  • Eval 中 LLM 偶有失败：跑 60 份真 PDF 时出现 3 次 LLM 连接错误（代理 hiccup），约 1% 噪声。
  • Ground truth 不完美：真 PDF 的 expected_HIGH 是按自动 skill 分桶决定的，可能存在标签噪声 10-15%。
  • LiveCareer 来源数据的 ToS 灰区：原始数据集已经过学术再分发，MIT licensed。生产部署绝对不要直接用，必须从 consent 候选人重新获取。
